# Итоговое описание проекта: Financial Owner Assistant

## Бизнес-проблема

Собственник малого и среднего бизнеса получает управленческую отчётность (БДР и ДДС) в Excel или PDF. Чтобы понять, что происходит в компании, ему нужен финансовый директор или бухгалтер-аналитик. Без них отчёт остаётся нечитаемым набором таблиц.

**Задача:** дать собственнику инструмент, который за секунды переводит сухие таблицы в понятный управленческий вывод — без финансового образования и без посредников.

---

## Решение

Трёхэтапный пайплайн на Python + OpenAI API:

1. **Извлечение данных** — парсер читает Excel и PDF, извлекает структурированные таблицы
2. **Детерминированный анализ** — нормализация, расчёт KPI и агрегации выполняются Python-кодом без LLM
3. **Генерация объяснения** — LLM получает только готовые цифры и формирует текст на человеческом языке

Принципиальное решение: **LLM не считает**. Все числа рассчитывает Python. LLM только объясняет.

---

## Что реализовано

### Ядро пайплайна
- Парсинг Excel (openpyxl) и PDF (pdfplumber) с извлечением таблиц БДР и ДДС
- Нормализация в единую финансовую модель (`pnl` + `cashflow`)
- Автоматическое определение отчётного периода (месяц/квартал/год)
- Автоматическое определение проектов и направлений бизнеса
- Расчёт управленческих KPI: выручка, операционная прибыль, валовая маржа, точка безубыточности, кассовый разрыв
- Рейтинг рентабельности проектов с расчётом proxy-прибыли

### Владельческий отчёт
- Генерация структурированного текстового отчёта (GPT-4o)
- Разделы: ключевые выводы, KPI, динамика по месяцам, анализ проектов, риски, рекомендации
- Отчёт привязан к конкретным числам из данных, не к общим фразам

### Q&A по отчёту *(экспериментальный модуль)*
- Трёхзвенная архитектура: Плановик → Исполнитель → Компоновщик
- Плановик классифицирует намерение и строит план запроса на правилах (LLM — запасной вариант)
- Исполнитель выполняет Python-обработчики, никогда не передавая сырые числа LLM
- Компоновщик (LLM) объясняет готовый payload
- Поддерживаемые типы вопросов: расходы (с фильтром по БДР/ДДС), рентабельность проектов (все / убыточные / прибыльные), KPI, тренды, точка безубыточности

## Ограничения

| # | Ограничение | Причина |
|---|------------|---------|
| 1 | Зависимость от структуры исходного файла | Парсер ожидает стандартные таблицы; сканы и нестандартные форматы дают неполный результат |
| 2 | Прибыль проекта — proxy | Расходы делятся пропорционально выручке; точный учёт требует раздельных данных |
| 3 | Чистая прибыль = операционная прибыль | Налоги на прибыль и внеоперационные статьи не выделяются |
| 4 | Q&A — базовые вопросы | Нестандартные, многошаговые и перекрёстные вопросы могут дать неточный ответ |
| 5 | Один период за раз | Сравнение с предыдущим периодом не реализовано |
| 6 | Без прогнозирования | MVP анализирует факт, не строит прогноз |

## Дальнейшее развитие (роадмап)

1. **Сравнение периодов** — план-факт, год к году, квартал к кварталу
2. **Улучшенный парсер** — поддержка нестандартных форматов и сканов (OCR)
3. **Точное распределение затрат** — учёт расходов по проектам из раздельного учёта
4. **Расширение Q&A** — многошаговые диалоги, перекрёстные запросы по нескольким периодам
5. **Уведомления** — автоматический алерт при пересечении пороговых значений KPI
6. **Мультиарендность** — изолированные сессии для нескольких компаний
