# Financial Owner Assistant

ИИ-ассистент для анализа управленческой отчётности

## 1. Краткое описание проекта

Рабочий MVP ИИ-ассистента для анализа управленческой отчётности малого и среднего бизнеса. Ассистент помогает собственнику бизнеса быстро понять данные БДР, ДДС и других управленческих отчётов: что произошло с бизнесом, где есть отклонения, какие расходы требуют внимания, есть ли риск кассового разрыва и что важно проверить в первую очередь.

## 2. Какую проблему решает проект

Собственники бизнеса часто не имеют времени и ресурсов на ручную разборку финансовых отчётов. Проект помогает сократить время на первичный анализ, выделить ключевые отклонения и подготовить управленческий вывод простым языком без необходимости вручную «читать» большие таблицы.

## 3. Что умеет MVP

- загрузка управленческих отчётов Excel и PDF;
- обработка БДР и ДДС;
- извлечение и нормализация финансовых данных;
- расчет ключевых KPI собственника;
- расчет proxy-точки безубыточности;
- анализ прибыли, расходов и денежных потоков;
- выявление ключевых отклонений и проблемных зон;
- объяснение разницы между прибылью и деньгами;
- формирование управленческого отчёта простым языком;
- ответы на уточняющие вопросы по рассчитанному отчёту;
- fallback-сообщения при неполных данных, ошибках обработки или вопросах вне рамок проекта.

## 4. Архитектура обработки данных

Ключевой принцип проекта: Python считает, LLM объясняет.

```text
PDF / Excel
→ парсинг
→ очистка таблиц
→ нормализация
→ canonical model
→ расчет KPI
→ analysis_context
→ управленческий отчет
→ Q&A по рассчитанному отчету
```

Исходные PDF и Excel не передаются напрямую в LLM для самостоятельного анализа. Сначала Python извлекает и структурирует данные, затем рассчитывает показатели и формирует расчетный контекст. После этого LLM получает уже подготовленные данные и объясняет их простым языком.

## 5. База знаний

В проекте используется локальная база знаний в Markdown/YAML.

```text
knowledge_base/
├── 00_role/
├── 01_report_templates/
├── 02_financial_methodology/
├── 03_mapping/
├── 04_language/
├── 05_industry_cases/
├── 06_quality_control/
└── 07_examples/
```

База знаний содержит:
- роль ассистента как финансового консультанта;
- шаблоны управленческих отчётов;
- методологию анализа БДР и ДДС;
- правила маппинга строк отчётности;
- правила языка ответа;
- отраслевые кейсы;
- правила контроля качества данных;
- примеры хороших управленческих отчётов.

## 7. Ограничения MVP

- до 10 файлов за один анализ;
- до 20 МБ на один файл;
- до 100 МБ общей загрузки;
- PDF — до 50 страниц на один файл;
- Excel — до 20 листов на один файл;
- поддерживаемые форматы: `.xlsx`, `.xls`, `.pdf`;
- история анализов не хранится;
- интеграции с 1С, банками и BI-системами не реализованы;
- Q&A-модуль является экспериментальным и лучше работает с вопросами по уже рассчитанному отчёту;
- полноценная RAG-система с embeddings, top-k поиском и отдельной векторной базой данных рассматривается как возможное развитие при масштабировании базы знаний.

## 8. Тарифная логика MVP

Коммерческая модель проекта: B2B SaaS + первичная консультационная настройка.

Базовая модель:
- первичная настройка: 25 000 ₽ разово;
- подписка: 15 000 ₽ в месяц за одну компанию;
- доступ предоставляется персонально для одной компании-клиента;
- в базовый тариф входит до 20 операций в месяц.

Операцией считается:
- формирование полного управленческого отчёта по загруженным файлам;
- один уточняющий вопрос пользователя по результатам уже рассчитанного отчёта.

После использования месячного лимита клиент может приобрести дополнительный пакет операций. Для действующих клиентов дополнительный пакет предоставляется со скидкой 20% от стандартной стоимости.

## 9. Экономика MVP

- 10 компаний-клиентов;
- 1–2 пользователя на компанию;
- около 15 активных пользователей в месяц;
- до 200 операций в месяц;
- расчетная стоимость API при 200 операциях: около $2.80 в месяц;
- плановый API-бюджет с запасом: до $50 в месяц;
- серверная инфраструктура: около 3 000–4 000 ₽ в месяц;
- общий бюджет первого года: 228 000 ₽ + до $600 API;
- операционная безубыточность: 1 клиент;
- окупаемость с учетом разработки и инфраструктуры: 2 клиента;
- срок выхода в плюс при 2 клиентах: около 7–8 месяцев.

## 10. Roadmap

- улучшение хранения расчетного контекста и метаданных;
- фиксация источника каждого показателя: файл, лист, период, статья, проект, тип отчёта;
- доработка Q&A-модуля и follow-up вопросов;
- улучшение работы с иерархией статей отчётности;
- экспорт отчёта в DOCX/PDF;
- личные кабинеты и история анализов;
- расширенные тарифы и учет лимитов;
- интеграции с 1С, BI-системами и банковскими выписками;
- возможный переход к полноценной RAG-системе при масштабировании базы знаний.

## 11. Запуск проекта

### Локальная разработка

```bash
python -m venv venv
venv\Scripts\activate
pip install -r requirements.txt
copy .env.example .env
uvicorn app.main:app --reload --port 8010
```

Приложение будет доступно по адресу [http://localhost:8010](http://localhost:8010).

### Docker / Docker Compose

```bash
docker compose up --build
```

### Переменные окружения

Минимально необходимые переменные в файле `.env`:

| Переменная | Назначение | Пример |
| --- | --- | --- |
| `OPENAI_API_KEY` | Ключ OpenAI API | `sk-...` |
| `OPENAI_MODEL` | Модель по умолчанию | `gpt-4o-mini` |
| `OPENAI_TEMPERATURE` | Температура генерации | `0.2` |

Дополнительные переменные:

| Переменная | Назначение | Дефолт |
| --- | --- | --- |
| `OPENAI_MODEL_FAST` | Быстрая модель для планировщика | `gpt-4o-mini` |
| `OPENAI_MODEL_STRONG` | Более сильная модель для отчёта | `gpt-4o` |
| `MAX_FILES` | Максимум файлов за сессию | `10` |
| `MAX_FILE_SIZE_MB` | Максимум размера одного файла | `20` |
| `MAX_TOTAL_SIZE_MB` | Максимум суммарного размера | `100` |

### Развёртывание на VPS/VDS

Подробная инструкция доступна в [docs/deployment_guide.md](docs/deployment_guide.md).

## 12. Безопасность

В публичный репозиторий не должны попадать:
- `.env`;
- API-ключи;
- загруженные пользовательские файлы;
- промежуточные данные парсинга;
- результаты анализа;
- виртуальное окружение;
- временные и служебные файлы.

Для этого используется `.gitignore`, который исключает пользовательские файлы, результаты анализа, виртуальное окружение и секреты.

## 14. Итог

Financial Owner Assistant — это рабочий MVP ИИ-ассистента для анализа управленческой отчётности, который помогает собственнику быстрее понимать финансовую ситуацию бизнеса, выявлять отклонения и принимать управленческие решения на основе структурированных данных и понятного объяснения.
